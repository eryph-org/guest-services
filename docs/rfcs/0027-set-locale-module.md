# RFC 0027 — `locale` / `keyboard` cloud-config module

Status: Implemented
Implemented in commit `<pending>`

## Problem

Cloud-init's `cc_set_locale` (`locale:`) and `cc_keyboard`
(`keyboard.layout:`) keys were silently dropped on egs-service-managed
catlets. Operators routinely need to flip a fresh catlet to a German /
French / Japanese display + input method without dropping into runcmd.

## What ships

### Schema

```yaml
locale: de-DE                # BCP-47 culture name
keyboard:
  layout: en-US              # BCP-47 (uses default layout for that culture)
  # or:
  layout: "0407:00000407"    # Windows KLID form (language:KLID)
```

Both keys are independent — keyboard-only is a valid scenario (operator
wants QWERTZ but leaves UI in English).

### Module

- `Eryph.GuestServices.Provisioning.Modules.SetLocaleModule`
  (`Stage.Network`, `Order = 5`, `Frequency = PerInstance`).
- Both fields absent → no-op.
- Locale-only:
  - `Set-Culture`
  - `Set-WinUILanguageOverride`
  - `New-WinUserLanguageList` + `Set-WinUserLanguageList`
  - `Set-WinSystemLocale` (only when current ≠ desired) — this is the
    "non-Unicode ANSI codepage" knob; changing it requires reboot.
- Locale+keyboard: the keyboard's `InputMethodTips` overrides the
  default layout the new language list would otherwise pick.
- Keyboard-only: `Get-WinUserLanguageList` (amend, not replace) → clear
  the first language's `InputMethodTips` → add the requested layout.
- Reboot semantics: when `Set-WinSystemLocale` had work to do the OS
  layer reports `RebootRequired`; the module returns
  `ModuleOutcome.RebootRequested`. Pure culture / UI-language /
  keyboard changes do not need reboot.

### OS seam

```csharp
Task<LocaleApplyResult> ApplyLocaleAsync(LocaleSpec spec, CancellationToken ct);
```

`LocaleSpec { Locale, KeyboardLayout }`. `LocaleApplyResult { RebootRequired }`.
The actual work is a single PowerShell script generated by
`WindowsOs.BuildLocaleScript` and executed via `powershell.exe -Command`.

`BuildLocaleScript` is `internal static` and has direct unit tests
(`BuildLocaleScriptTests`) — the script generation is a composition
root that's awkward to exercise end-to-end without a real Windows
guest.

## Why one module for two cloud-init keys

Cloud-init separates `cc_set_locale` and `cc_keyboard` because Linux
handles the two via distinct files / packages. On Windows the language
list and input method are coupled: `New-WinUserLanguageList` returns a
list whose first entry carries `InputMethodTips`. Splitting would force
the keyboard module to re-fetch and mutate state the locale module just
wrote, producing race-prone behaviour.

## What we deliberately do NOT do (vs cloud-init)

- No `Set-WinHomeLocation` — would need a BCP-47 → GeoID lookup table
  that drifts over time. Operators set this via `runcmd` if needed.
- No keyboard `model` / `variant` / `options` — these are X11 concepts
  with no Windows analogue. The simplified `KeyboardConfig { Layout }`
  is intentional.
