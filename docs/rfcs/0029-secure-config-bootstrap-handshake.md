# RFC 0029 — Secure config delivery via bootstrap handshake

Status: Draft

## Problem

Catlet user-data routinely needs to carry secrets — admin passwords,
join tokens, license keys, SaaS API credentials. On every supported
target the plaintext user-data is at-rest readable:

| Cloud | User-data lands at | Confidentiality |
|---|---|---|
| Azure | `C:\AzureData\CustomData.bin` | Plaintext on disk; MS docs explicitly warn against secrets |
| AWS | IMDS user-data endpoint | Plaintext, readable by any in-VM process that can hit IMDS |
| GCP | Metadata user-data | Same |
| OpenStack / Hyper-V (local eryph) | ConfigDrive ISO | Plaintext on the drive, readable by host operator |

We need a delivery shape that lets eryph cloud (the scheduler) hand a
secret-bearing cloud-config to a specific catlet **without that secret
ever appearing in user-data, on disk, or over an attacker-readable
channel** — and that works the same way on every target cloud.

Scope is bounded to "eryph cloud ↔ the catlet it just provisioned".
This is NOT general-purpose secret management — no KMS integration, no
shared-vault concepts, no key rotation across many consumers. The
two-party model means we can use direct asymmetric crypto rather than a
trust-broker architecture.

## What cloud-init does

Nothing for this case. The `password:` and `chpasswd:` blocks accept
plaintext only. The convention is "fetch secrets from a cloud-native
KMS at boot" (Azure Key Vault + managed identity, AWS Secrets Manager
+ IAM role, GCP Secret Manager + service account). That works
per-cloud but is a separate integration per target — exactly what we're
trying to avoid.

## What cloudbase-init does

Nothing comparable. Azure-image cbi relies on PA's OSProfile cert
mechanism for `AdminPassword` and SSH keys — Azure-only, fabric-issued
cert, decrypted by PA during OOBE. Not portable.

## What ships

A two-stage delivery: plaintext user-data carries only **handshake
setup**; the agent uses it to obtain a per-VM, one-time include URL
serving the real (encrypted) cloud-config.

### Top-level cloud-config key

```yaml
#cloud-config
secure_bootstrap:
  endpoint: https://eryph.example/v1/bootstrap
  server_pubkey: |
    -----BEGIN PUBLIC KEY-----
    MIIBIjANBgkqhkiG9w0BAQEFAAOCAQ8AMIIBCgKCAQEA...
    -----END PUBLIC KEY-----
  bootstrap_token: 7s9F2k8nQ3aP1r5tYwXjZbVcN6dHmL0e
  cipher: rsa-oaep-sha256        # or x25519-chacha20-poly1305
```

| Field | Required | Purpose |
|---|---|---|
| `endpoint` | yes | HTTPS URL the agent POSTs to during step (2) of the protocol. eryph cloud's bootstrap service. |
| `server_pubkey` | yes | eryph cloud's long-lived public key. The agent pins it for server authentication so a network attacker cannot impersonate eryph cloud. PEM-encoded SubjectPublicKeyInfo. |
| `bootstrap_token` | yes | Single-use, per-catlet credential generated by eryph cloud at scheduling time. eryph cloud burns it on first redemption. Authenticates the agent as "the catlet I issued this token for". |
| `cipher` | yes | Negotiated cipher suite name. v1 supports `rsa-oaep-sha256` and `x25519-chacha20-poly1305`. Module rejects unknown values rather than silently downgrading. |

`secure_bootstrap` is eryph-namespaced because the shape is
eryph-specific. Added to the acknowledged-key inventory in
`CloudConfigSerializer` as an eryph-extension so a cross-OS user-data
does not warn-spam on the side that does not natively consume it.
(Linux runs the same logic via cloud-init's `text/part-handler` — see
"Cross-OS" below.)

### Handshake protocol

```
agent (VM)                                   eryph cloud
  |                                              |
  | (1) generate ephemeral VM keypair            |
  |     (kp_vm = {pk_vm, sk_vm})                 |
  |                                              |
  | (2) POST endpoint                            |
  |     body = enc(server_pubkey, {              |
  |       bootstrap_token,                       |
  |       pk_vm,                                 |
  |       nonce: <fresh 16 bytes>                |
  |     })                                       |
  |                                              |
  |     headers:                                 |
  |       X-Eryph-Bootstrap: v1                  |
  |                                              |
  |--------------------------------------------->|
  |                                              |
  |                                  verify bootstrap_token
  |                                  burn token (atomic, single-use)
  |                                  generate per-VM include URL (one-time signed)
  |                                              |
  | (3) response                                 |
  |     body = enc(pk_vm, {                      |
  |       include_url,                           |
  |       nonce_echo: <same 16 bytes>            |
  |     })                                       |
  |<---------------------------------------------|
  |                                              |
  |     verify nonce_echo matches request nonce  |
  |     decrypt include_url                      |
  |                                              |
  | (4) GET include_url                          |
  |--------------------------------------------->|
  |                                              |
  |                                  verify URL signature, expiry
  |                                              |
  | (5) response                                 |
  |     body = enc(pk_vm, {                      |
  |       cloud_config_fragment: <YAML>          |
  |     })                                       |
  |<---------------------------------------------|
  |                                              |
  |     decrypt, merge fragment into current     |
  |     cloud-config, continue normal pipeline   |
  |                                              |
  | (6) discard sk_vm (zeroize)                  |
  |                                              |
```

Properties this gives us:

- **Bootstrap token only useful once.** Replay after burn is rejected.
- **VM verifies it's talking to eryph cloud** before sending `pk_vm` — `server_pubkey` is pinned in user-data; an attacker who intercepts the callback URL cannot forge a valid response because they don't hold `sk_server`.
- **eryph cloud verifies the agent is the catlet it scheduled** via `bootstrap_token`. Any other holder of the token (image-thief, snapshot-leak) gets exactly one shot, and the shot is the legitimate catlet's first boot — race-controllable by burning the token at scheduling-completion rather than at first-redemption.
- **Cloud-config fragment encrypted end-to-end to `pk_vm`.** TLS is belt-and-suspenders; even a TLS-MITM cannot read the plaintext.
- **Ephemeral keypair, discarded post-decrypt.** No long-lived secret on the VM to exfiltrate. Forward secrecy: if `sk_server` leaks later, recorded traffic from past bootstraps is still unreadable.

### Module

- `Eryph.GuestServices.Provisioning.Modules.SecureBootstrapModule`
  (`Stage.Local`, `Order = 0`, `Frequency = PerInstance`).
- Runs **first** in Local — before any module that consumes secrets
  (`UsersGroupsModule`, `SshModule`, license, etc.) — so the decrypted
  fragment is merged into the cloud-config before downstream modules
  read it.
- Returns `ModuleOutcome.Failed` (not Completed) on handshake failure
  — without the secret fragment, downstream provisioning produces a
  partially-configured VM that may silently ship without the operator's
  password. Fail-closed is the only safe outcome.

### Network seam

- `IBootstrapClient.ExchangeAsync(SecureBootstrapConfig, CancellationToken)`
  returns the decrypted cloud-config fragment as `byte[]`.
- Production impl uses the existing `IUrlHelper` HTTPS stack
  (cert pinning, retry/backoff policy from `ProvisioningSettings`).
- Test seam: `Func<SecureBootstrapConfig, CancellationToken, Task<byte[]>>`
  injected the same way `AzureImdsClient` is, to exercise the protocol
  end-to-end without a network.

### Crypto

v1: two suites, both well-supported in .NET's BCL.

| Suite | `cipher:` value | Notes |
|---|---|---|
| RSA-OAEP / SHA-256 | `rsa-oaep-sha256` | Server key is 3072-bit RSA. VM keypair generated via `RSA.Create(3072)`. Simplest interop. |
| X25519 + ChaCha20-Poly1305 | `x25519-chacha20-poly1305` | Smaller payloads, modern. Server key is X25519 public point. Requires `BouncyCastle` (already a transitive dependency of `DevTunnels.Ssh.Extensions`). |

No DH "negotiation" — the cipher is fixed per `secure_bootstrap.cipher`
field. Mixed-cipher mid-protocol is rejected.

## Cross-OS

The same `secure_bootstrap:` block is consumed on Linux by a
**cloud-init `text/part-handler` Python module** shipped as part of
the eryph fodder, not by us:

- eryph emits a multipart user-data: `text/part-handler` part (the
  Python handler) + `text/cloud-config` part (the `secure_bootstrap:`
  block + any plaintext config).
- cloud-init loads the part-handler, dispatches the cloud-config part
  to it, the Python code runs the same handshake protocol, decrypts,
  and writes the result to `/var/lib/cloud/seed/nocloud/user-data` so
  cloud-init re-reads it.
- Reuses cloud-init's own extension mechanism (RFC 0012 documents this
  shape on the Windows side as "deferred indefinitely"; on Linux it's
  cloud-init-native and free).

Symmetric outcome: Windows runs the handshake in egs; Linux runs it in
cloud-init via part-handler. Both consume the same wire schema. The
Python part-handler implementation is out of scope for this RFC — it's
authored once in the eryph-fodder repo, not in guest-services.

## Threat model

Explicit about what this DOES and DOES NOT protect:

| Adversary | Protected? |
|---|---|
| Network observer (TLS interceptor included) | YES — fragment is encrypted to `pk_vm`, not to TLS |
| Anyone who reads the catlet's user-data after the fact (disk snapshot, host operator, Azure portal "OS profile" view) | YES — user-data contains only the handshake setup; the secret fragment is fetched and decrypted in-memory |
| Anyone who reads user-data AND can race the legitimate catlet to redeem the bootstrap token | PARTIALLY — token is single-use, so the attacker can only win the race ONCE per catlet, and only if they win before the catlet does. eryph cloud SHOULD set a tight expiry (minutes) and SHOULD log every redemption for operator-side alerting. |
| Compromised eryph cloud (`sk_server` leak) | NO — eryph cloud can decrypt the bootstrap request and read `pk_vm`. But forward secrecy holds: ephemeral `kp_vm` means past sessions are still safe; only future sessions are compromised until rotation. |
| In-VM unprivileged user during the window between fetch and pipeline-end | NO — the decrypted fragment sits in agent memory and (depending on which downstream module consumes it) may transit through other agent state. Out of scope; existing OS user model applies. |
| Compromised catlet (attacker-controlled VM consumes the bootstrap token) | NO BY DESIGN — eryph cloud's threat model says the catlet it scheduled IS trusted. If the image is malicious, all bets are off. |

## Fallback for airgapped / no-network-at-first-boot deployments

The bootstrap handshake requires the VM to reach `endpoint` over the
network during Local stage. For air-gapped or network-late deployments
this is not possible. Two fallback shapes are available; both are
described here for completeness but are NOT v1 deliverables:

1. **Offline-mount key injection.** eryph cloud mounts the catlet's
   VHD before first boot, writes both the encrypted fragment AND the
   per-catlet private key into a known path. The module reads them
   instead of doing the handshake. Network independent; requires
   eryph cloud to have filesystem access to the VHD pre-boot. (Already
   matches the e2e provisioning suite's offline-mount workflow.)
2. **KVP-delivered key for local Hyper-V.** When the deploy target is
   eryph-local Hyper-V (same hypervisor as build), the per-catlet key
   can be delivered via the existing KVP channel rather than over
   network. Cross-cloud is excluded by definition.

Selection: `secure_bootstrap.endpoint` absent + presence of an
alternative fallback key in the same block (`secure_bootstrap.offline_key_path`
or `secure_bootstrap.kvp_key`) switches the module to the fallback
shape. v1 fails closed on missing endpoint; fallbacks land in later
RFCs once the v1 protocol is settled.

## Open questions

- **Cipher default.** `rsa-oaep-sha256` is the lowest-friction option
  (no `BouncyCastle` pull-in on the Windows side, native `RSA.Create`).
  `x25519-chacha20-poly1305` is smaller and faster but adds a dep.
  Lean RSA for v1; add X25519 in v1.1 when there's a demand signal.
- **Token lifetime.** What's the right default expiry for
  `bootstrap_token`? Too short = legitimate catlet misses its window
  during a slow Azure PA pass; too long = race attacker has more time.
  Initial proposal: 30 minutes, with the catlet's `secure_bootstrap`
  block also carrying an `expires_at` field eryph cloud sets at issue.
- **Replay reporting.** Should eryph cloud log every token redemption
  attempt (including the burned-second-attempt rejects) for operator
  alerting? Probably yes — a second redemption is either a retry on a
  partial first attempt (benign, but should be flagged) or a race
  attacker (urgent). Out of scope for this RFC; tracked as a
  cloud-side concern.
- **What about `chpasswd:` users coming in via the fragment?** The
  decrypted fragment is fed into the same `CloudConfigMerge` path as
  the plaintext config — so `users:`, `chpasswd:`, `runcmd:`, etc.,
  all work transparently. Validation order matters: the secure
  fragment merges BEFORE `CloudConfigValidations` runs, so the merged
  config is a single validate-target.
- **Multipart payload handling on Linux.** The part-handler approach
  requires the cloud-config part to be MIME-attached, not the root
  user-data. eryph fodder must emit multipart for the Linux path. The
  Windows path is happy with either (single cloud-config OR multipart
  with the secure block in one part). Worth pinning the multipart
  shape in a sister RFC under the fodder repo.
- **Retry policy.** Bootstrap endpoint unreachable at first boot:
  `WaitForReady`-style retry with exponential backoff (mirror RFC 0004
  pattern) or hard-fail immediately? Lean retry — same shape as
  `AzureDataSource`'s PA-readiness wait — capped at the
  `DataSources.ReadinessTimeoutMinutes` budget. Decided in
  implementation, not RFC.
